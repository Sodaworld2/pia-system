1. What Is a Soul?

Every agent in PIA has a soul — a JSON file that defines who they are. It is loaded from disk, seeded into the database on startup, and injected into every Claude AI session as the system prompt. The soul is what makes Fisher2050 different from a generic AI assistant. Without it, you have a language model. With it, you have a named team member with a personality, goals, working style, and defined relationships.

Think of it like a permanent briefing document that Claude reads before every conversation. "You are Fisher2050. Here is your personality. Here is your job. Here is who you work with. Here is how you operate." That document is the soul.

The key insight Souls live in src/souls/personalities/*.json. You do not need to write code to change an agent's behaviour. You edit the JSON. The next time that agent is spawned, it reads the updated soul. Non-technical editing, version-controllable, portable.

There are currently 12 souls written and stored. A 13th (Itai) is planned. All 12 are seeded into the SQLite database every time PIA starts.

2. Anatomy of a Soul — The 7 Fields

id
Unique identifier (slug) The machine-readable name. Used in the database, the API, and Fisher2050's scheduling. Must be lowercase, no spaces. e.g. "fisher2050", "tim_buc", "bird_fountain"
name
Display name The human-readable name shown in the dashboard and in briefings. e.g. "Fisher2050", "Tim Buc", "Bird Fountain"
role
One-line job title Appears under the name. Describes the function in plain English. e.g. "Project Manager and Machine Resource Scheduler"
email
Agent email address (optional) The digital identity for human-facing agents. Used for sending briefings, receiving tasks, signing off on deliveries. Infrastructure agents (Owl, Monitor, Coder Machine) have no email — they don't communicate with humans directly. e.g. "fisher2050@sodalabs.ai"
personality
Character definition (500–1900 characters) Prose that defines who this agent IS — their communication style, personality traits, and behavioural quirks. This is what makes the agent feel like a specific person rather than a generic AI. Include: communication style, traits (3–5), quirks (2–4 specific habits or phrases they use). The more specific the quirks, the more consistent the personality.
goals
What success looks like (3–6 items) Measurable outcomes the agent is working toward. Not tasks — outcomes. Written as statements that could be scored "met/not met." These orient the agent's decision-making when the instruction is ambiguous. e.g. "Ziggi scores Farcake output at 8/10 or above on average"
relationships
Who this agent works with (3–11 entries) Key/value pairs: agent name → what the relationship is and how it works. This defines the org chart in the soul. An agent that knows Ziggi is its quality gate will automatically hold work for review. An agent that knows Tim Buc is the archive destination will file there. Relationships are the wiring instructions baked into the personality.
system_prompt
Operational instructions (300–1900 characters) The step-by-step process this agent follows every time it runs. Starts with Mic's working style rules (same in every agent), then the specific numbered process. This is the "how to do the job" layer. Personality says who they are; system_prompt says what they do, in what order, every time. The more precise the steps, the more consistent the output.
config
Technical configuration Machine assignment, trigger type, scheduling times, thresholds. Tells the system how and when to spawn this agent. Key fields: machine (M1/M2/M3/dedicated), trigger (cron/event/calendar_entry), ephemeral (true = spawns and terminates, false = always on), always_on, quality_threshold.

The "Mic Preamble" — In Every Single Soul

The first lines of every agent's system_prompt are identical:

[Mic's Working Style]
- Lead with the answer, reasoning after
- Max 3 key points at once
- Never re-explain what Mic already knows
- Surface time-sensitive items first
- Push back if something is wrong — don't validate for the sake of it

This is non-negotiable. Every agent in the fleet communicates in Mic's preferred style, even when talking to other agents. It propagates Mic's preferences through the entire system automatically.

3. How Souls Are Programmed

The Rules for Writing a Good Soul

FieldWhat Makes It GoodCommon Mistake
personality Specific quirks with exact phrasing. "Ends every report with: 'Filed: X sessions.'" Not just "methodical and reliable." Generic adjectives with no observable behaviours. "Helpful, efficient, professional." This produces inconsistent output.
goals Scoreable outcomes. "Ziggi scores output at 8/10 or above." Fisher2050 can use this to decide if re-work is needed. Vague aspirations. "Do good work." Nothing in the system can act on this.
relationships Defines the flow of work. "Ziggi: Mandatory quality gate. Holds every draft." The agent internalises this as behaviour. Missing relationships. If an agent doesn't know Tim Buc is the archive destination, it won't file there.
system_prompt Numbered steps. Specific format templates. "Verdict format: Ziggi's Verdict: [score]/10. Issue: [description]." Prose instructions without structure. Agents drift when steps aren't numbered and output formats aren't explicit.
config.ephemeral true for task agents (Farcake, Andy, Ziggi). They spawn, run one task, terminate. Cheap and focused. Setting ephemeral to false for task agents. They'll accumulate context and eventually hit limits without completing work.

Documentation Limits — How Much Can You Put In a Soul?

The soul is injected as a system prompt. Claude has a context window — the soul competes with the task input and the conversation history for that space. Guidelines:

FieldMinimumGood TargetHard Limit
personality200 chars800–1200 chars~2000 chars — beyond this it becomes noise
system_prompt300 chars800–1500 chars~2000 chars — keep steps tight
goals3 items5–6 items8 items max — more dilutes focus
relationships3 entries5–8 entries12 entries max
Total soul—3000–5000 chars8000 chars — above this eats agent context budget
The Soul Wiring Gap (Critical — February 2026) All 12 souls are written, seeded, and stored in the database. But agent-session.ts currently does NOT call generateSystemPrompt() before spawning. Every agent running today is running WITHOUT their soul injected. This is a single ~20-line fix. Until it is applied, agents behave as generic Claude assistants, not as Fisher2050 or Eliyahu.

7. What's Missing — The Wiring Gap

The Most Important Fix in the Entire System

All 12 souls exist. The engine that assembles them into a system prompt exists (generateSystemPrompt() in soul-engine.ts). The seeding runs on startup. But agent-session.ts — the file that actually spawns every agent — never calls generateSystemPrompt().

Result: every agent runs as a generic Claude assistant. Fisher2050 does not know it is Fisher2050. Ziggi does not know to score 1–10. Andy does not know to ask for voice samples.

The fix is approximately 20 lines in src/mission-control/agent-session.ts. Before spawning, call generateSystemPrompt(agentId) from the soul engine and inject it as the system prompt. This single fix activates the entire personality layer across all 12 agents simultaneously.
