🧠 Agent Core
The fundamental building blocks of an AI agent in PIA
Soul CORE
A persistent AI personality definition. Contains identity, personality description, goals, relationships, config, and system prompt. Stored as JSON file, loaded into SQLite database at startup. Survives across sessions and machines.
// Defined in: src/souls/personalities/fisher2050.json { "id": "fisher2050", "name": "Fisher2050", "role": "Project Manager", "personality": "You are Fisher2050, organized, proactive...", "goals": ["Keep all projects on track..."], "relationships": {"Mic": "The boss..."}, "system_prompt": "When processing a task: 1. Assess...", "config": {"dailyReviewTime": "09:00"} }
Files: src/souls/personalities/*.json src/souls/soul-engine.ts
Personality CORE
The detailed multi-paragraph description of how an agent thinks, communicates, and behaves. Includes communication style (e.g., "Direct and action-oriented"), personality traits (e.g., "Methodical, Persistent, Pragmatic"), and quirks (e.g., "Ends updates with confidence %"). This is what makes each agent unique and consistent.
// Fisher2050's personality (~900 chars): "Your communication style: - Direct and action-oriented - You use bullet points and clear structure - You follow up on outstanding items without being asked Your quirks: - You end status updates with a confidence % ('Project health: 78%') - You refer to the team as 'the crew'"
Goals CORE
Array of specific objectives the agent actively pursues during every task. These are injected into the system prompt and guide the agent's priorities. Typically 4-6 goals per soul.
// Eliyahu's goals: [ "Process all session journals, logs, and work output daily", "Send morning knowledge briefings with yesterday's insights", "Track decisions and their outcomes across all projects", "Ask probing questions that surface hidden problems", "Maintain a cross-project knowledge base of patterns", "Send weekly 'State of the System' reports" ]
Relationships CORE
A map defining how a soul relates to other souls and humans. Affects tone, deference, and collaboration patterns. Each relationship entry describes the dynamic in plain language.
// Ziggi's relationships: { "Mic": "The boss. Respects Mic's vision, translates into solid architecture. Pushes back gently on tech debt.", "Fisher2050": "The PM. Triggers Ziggi for post-session reviews. Reports findings back for task tracking.", "Eliyahu": "The Knowledge Manager. Shares architectural insights for cross-referencing.", "Machine Agents": "The build environments. Knows what each is good at." }
System Prompt CORE
Operational instructions for the agent. Defines what it CAN access (database, email, orchestrator, other agents), what steps to follow for each task type, and behavioral boundaries. This is the "instruction manual" layer.
// Ziggi's system_prompt: "When triggered for a post-session review: 1. Check git log for recent changes 2. Read the changed files 3. Evaluate: code quality, naming, patterns, potential issues 4. Update: functional spec, technical spec, architecture docs 5. Report findings to Fisher2050 and Mic For deep reviews (2am): 1. Review entire codebase structure 2. Check for technical debt accumulation 3. Browse web for dependency updates"
Config CORE
JSON object of timing, thresholds, and behavior switches. Controls when scheduled tasks run, quality minimums, feature flags, and operational limits.
// Fisher2050: // Ziggi: { "dailyReviewTime": "09:00", { "deepReviewTime": "02:00", "eveningSummaryTime": "18:00", "codeQualityMinimum": 7, "followUpIntervalHours": 4, "maxFilesPerReview": 50, "maxTasksPerAgent": 5, "trackTechnicalDebt": true, "confidenceThreshold": 0.7 } "useContext7": true }
System Prompt Generation CORE
The dynamic function generateSystemPrompt(soulId, extraContext) that assembles the full prompt Claude receives. Combines: identity + personality + goals + relationships + 20 recent memories + 10 important memories + task context. Regenerated fresh for every task.
Files: src/souls/soul-engine.ts line ~190
Seed Souls CORE
Startup function that loads personality JSON files from src/souls/personalities/ and upserts them into the SQLite database. Creates souls if new, skips if already exist. Runs automatically when PIA starts.
Files: src/souls/seed-souls.ts
Briefing Document CORE
A long markdown file (10-20KB) giving an agent complete project context. Contains: tech stack, database schemas, API routes, build plan with phases, critical rules, current status, and the Zero Questions Policy. Optional — loaded for complex tasks.
// DAO_AGENT_BRIEFING.md includes: - 22 database tables with full schemas - 21 API routes with descriptions - 6-phase build plan - Critical rules ("DO NOT stop the running DAO server") - Zero Questions Policy // Size: ~18KB = ~4,500 tokens
Files: DAO_AGENT_BRIEFING.md dao-foundation-files/VIDEOHOHO_AGENT_BRIEFING.md
🪠 Memory System
How agents learn, remember, and forget
Memory MEMORY
A stored record of something an agent experienced, decided, learned, or observed. Has: soul_id, category, content (text), importance (1-10), context (optional), is_summarized flag, and timestamp. Memories are the "training data" that accumulates from real work.
// Example memories: [experience] "Fixed 5 failing tests by updating mock library" (importance: 7) [decision] "Chose TypeScript over Python for DAO backend" (importance: 8) [learning] "Snapshot tests break if regenerated without review" (importance: 6) [observation] "CI pipeline slow on Fridays — batch jobs overlap" (importance: 4)
Memory Category MEMORY
One of 7 types that classify what kind of knowledge a memory represents. Affects how memories are grouped, surfaced, and summarized.
experience — Things done, outcomes, what worked/failed decision — Choices made, reasoning, alternatives considered learning — New knowledge, patterns discovered interaction — Communication with other agents, feedback received observation — Environmental changes, metrics, status updates goal_progress — Milestones reached, progress on long-term goals summary — Auto-generated compressed versions of old memories
Importance Score MEMORY
1-10 integer rating on a memory. Determines how long it persists and whether it's included in the "important memories" section of system prompts. Default: 5.
1-3: Low — Trivia, minor observations. Pruned after 30 days. 4-6: Medium — Normal decisions, regular interactions. Summarized after 7 days. 7-10: High — Breakthroughs, critical failures, key achievements. Stays FOREVER. // Important memories (>= 7) are ALWAYS loaded into system prompt // Up to 10 important memories included per task
Memory Summarization MEMORY
Automatic process that groups old memories (>7 days) by category and creates compressed summary memories. Original memories are marked is_summarized=1 (kept but excluded from active prompts). Prevents infinite context growth while preserving knowledge.
// Before summarization (5 separate memories): [experience/Feb 5] "Fixed auth bug in login route" [experience/Feb 5] "Updated user validation middleware" [experience/Feb 6] "Refactored session handling" [experience/Feb 6] "Added rate limiting to API" [experience/Feb 7] "Fixed CORS headers for frontend" // After summarization (1 summary memory): [summary/Feb 13] "Feb 5-7: Fixed auth/session bugs, updated middleware, added rate limiting, fixed CORS. 5 backend improvements."
Files: src/souls/memory-manager.ts summarizeOldMemories()
Memory Pruning MEMORY
Deletion of old, low-importance, already-summarized memories. Only targets memories that are: older than 30 days AND importance ≤ 3 AND already summarized. High-importance memories are never pruned.
Files: src/souls/memory-manager.ts pruneMemories(soulId, 30, 3)
Soul Engine MEMORY
The core TypeScript class that manages all soul operations: CRUD, memory access, system prompt generation. Singleton pattern — one instance shared across the application.
Files: src/souls/soul-engine.ts
Memory Manager MEMORY
The class handling all memory operations: add, get recent, get important, get by category, search (full-text), get stats, summarize old, prune old, delete specific.
Files: src/souls/memory-manager.ts
Soul Interaction MEMORY
Logged communication between two souls. Types: task_request, review_request, status_update, question, response, notification. Stored in soul_interactions table with from/to soul IDs, content, and metadata.
// Example interaction: { from_soul_id: "fisher2050", to_soul_id: "ziggi", interaction_type: "review_request", content: "Post-session review needed for DAO backend changes", metadata: { "project": "DAOV1", "files_changed": 12 } }
Decision Log MEMORY
Eliyahu's practice of tracking every decision made across projects: what was decided, why, what alternatives were considered, and what the outcome was. Stored as memories with category "decision".
⚙ Orchestration
How work gets distributed, executed, and tracked
Orchestrator ORCH
The system that distributes work across agents and machines. Three components: Task Queue (priority-based work list), Execution Engine (polling loop), and AI Router (model selector).
Files: src/orchestrator/ src/api/routes/orchestrator.ts
Autonomous Worker ORCH
The tool loop that sends tasks to Claude API with tool definitions and executes tool calls until the job is done. Tools available: run_command, read_file, write_file, list_directory, report_progress. Supports soul injection via soulId parameter.
// Flow: POST /api/orchestrator/run { task: "Fix tests", soulId: "ziggi" } -> Worker loads Ziggi's soul (personality + memories) -> Sends to Claude API with tool definitions -> Claude: "use tool: run_command('npm test')" -> Worker executes, returns result -> Claude: "use tool: write_file('test.ts', ...)" -> Worker writes file -> Loop until Claude says "done" -> Save completion memory to Ziggi's soul
Files: src/orchestrator/autonomous-worker.ts
Tool Loop ORCH
The cycle where Claude receives a task, calls a tool, gets the result, calls the next tool, and repeats until complete. Each iteration is one "turn". Bounded by maxTurns and maxBudget.
Task Queue ORCH
Priority-based (1-5, default 3) list of pending work with dependency tracking. Tasks flow: pending -> in_progress -> completed/failed. Supports blocking (task A must finish before task B starts).
Files: src/orchestrator/task-queue.ts
Execution Engine ORCH
Polling loop (every 5 seconds) that picks the highest-priority unblocked task, classifies its complexity, routes it to the appropriate AI model, executes it, records cost, and updates status.
Files: src/orchestrator/execution-engine.ts
AI Router ORCH
Cost-optimized model selector. Routes tasks to the cheapest model that can handle them. Tier 0: Ollama (free, local GPU). Tier 1: Claude Haiku (cheap). Tier 2: Claude Sonnet (smart). Falls back up the chain if primary fails.
Security tasks -> Always Sonnet (best reasoning) Simple tasks -> Ollama if available, else Haiku Medium/Complex -> Sonnet Fallback chain -> Ollama failed? Try Haiku. Haiku failed? Try Sonnet.
MaxTurns ORCH
Maximum number of tool-call rounds before the autonomous worker stops. Prevents infinite loops. Default: 30 turns.
MaxBudget ORCH
Maximum USD spend allowed for a single autonomous task. Prevents runaway API costs. Default: $1.00.
Work Session ORCH
A tracked period of human or agent work on a project. Records: start/end time, machine, project path, git branch, commits before/after, files changed, notes. Ending a session with changes auto-triggers Ziggi review.
Files: src/api/routes/work-sessions.ts
Auto-Review Trigger ORCH
When a work session ends and files_changed > 0, the system automatically spawns an autonomous task with soulId "ziggi" to perform a code review. No human action required.
Cost Waterfall ORCH
Routing strategy: try FREE (Ollama local) first, then CHEAP (Haiku), then SMART (Sonnet) only when needed. Minimizes API spend while maintaining quality.
Night Shift ORCH
The concept of agents working autonomously on scheduled tasks while humans sleep. Ziggi does deep code reviews at 2am. Eliyahu prepares morning briefings at 6am. Fisher2050 runs daily standup at 9am and evening summary at 6pm.
Context Injection ORCH
The act of loading a soul's personality + memories + goals + relationships + task context into Claude's system prompt before executing a task. This is what makes the agent "be" its soul.
🏗 Infrastructure
Machines, networks, context windows, and storage
Hub INFRA
The central PIA server (Machine #1, izzit7, 100.73.133.3) that coordinates all machines and agents. Runs on port 3000. All other machines connect to it via Tailscale.
Alive Repo INFRA
A git repository with a .pia/ directory containing: identity.json (who), state.json (current status), knowledge/ (brain), jobs/history.jsonl (memory), queue/pending.json (inbox), hooks/ (how), log/ (diary). Makes any repo a living agent.
.pia/ identity.json # WHO: name, capabilities, tech stack state.json # CURRENT: status, current task knowledge/ # BRAIN: everything repo knows about itself jobs/history.jsonl # MEMORY: every job ever executed queue/pending.json # INBOX: tasks waiting hooks/ # HOW: instructions for handling tasks log/ # DIARY: daily activity logs
Context Window INFRA
The total token budget Claude has for a conversation. Standard: 200,000 tokens (~150K words). Extended (beta): 1,000,000 tokens. 1 token = ~4 characters. The soul system uses ~4-8K tokens (2-4%), leaving 96%+ for work.
Auto-Compaction INFRA
Claude Code's automatic summarization of old conversation turns when context reaches ~83% capacity (~167K tokens). Older messages get compressed. Soul/personality data in the system prompt is NOT affected.
Heartbeat INFRA
Service that maintains machine registration and liveness. Reports every 30 seconds: CPU/memory/disk stats, agent count, system resources. If heartbeat stops, machine is marked offline.
Files: src/orchestrator/heartbeat.ts
PTY Session INFRA
Pseudo-terminal sessions that allow remote command execution on machines. Claude instances on different machines can control each other via PTY. Used for running builds, tests, and server management on Machine #3.
📚 Patterns & Rules
Best practices, policies, and anti-patterns
Zero Questions Policy PATTERN
Rule that agents must NEVER ask questions. They must decide autonomously, execute immediately, and report only after completion. Asking a question = task failure. Applied across all briefing documents.
NEVER: - "Should I...?" "Can I...?" "May I...?" - "I will..." "I'm going to..." - Preview actions or pause for approval ALWAYS: - Make the decision yourself - Execute immediately - Report results AFTER completion
Living Document PATTERN
A file that gets continuously updated across sessions. The source of truth for ongoing work. Examples: SESSION_JOURNAL.md, CLAUDE.md, tasks/master.md. Must be read at session start and updated at session end.
Ziggi's Verdict PATTERN
The final summary section of every Ziggi code review. Contains a code quality score (1-10) and recommendation. Ziggi's signature format that makes reviews consistent.
Architecture Smell PATTERN
Ziggi's term for structural issues that feel wrong even if tests pass. Similar to "code smell" but at the system/architecture level. Flagged in reviews as items to track or refactor.
Ghost Code PATTERN
Hallucinated changes that exist in Claude's context but were never actually committed to disk. Caused by long sessions where Claude "remembers" writing code that was never saved. Detected with git status and git diff.
Context Rot PATTERN
Accumulated conversation history that pollutes Claude's understanding of current state. Happens in long sessions with many file edits. Solution: one task per session, commit and start fresh.
Continuous Memory Collection PATTERN
The rule that agents must capture ALL decisions, ideas, research, bugs, file changes, questions, and preference changes as memories. "If you discussed it, write it down. If you built it, log it. If you decided it, record it. No exceptions."
Session Journal PATTERN
Chronological markdown document recording everything accomplished in each work session: tasks completed, decisions made, issues found, files modified, next steps. Must be updated at end of every session.
Files: JOURNAL_2026-02-13.md SESSION_JOURNAL_*.md
🛠 Template System (Martin's)
Alternative agent architecture from the orchestration template
Hook TEMPLATE
A shell script that runs before/after tool calls. Used to enforce rules: block dangerous commands (rm -rf), enforce delegation (prevent direct code edits), track costs, log changes. Types: PreToolUse, PostToolUse, SessionStart, SubagentStart/Stop.
Skill TEMPLATE
A defined capability stored as a SKILL.md file in .claude/skills/. Invoked via /skill-name. Contains instructions and workflow for a specific task type. Examples: /onboard, /debug, /create-hook, /consider:pareto.
Agent (Template) TEMPLATE
In Martin's template: a .md file in .claude/agents/ defining a specialized sub-agent. Has: name, description, tools available, model override, and full instructions. Examples: @local-coder, @debug, @code-sentinel, @overseer, @janitor.
Teaming Stub TEMPLATE
Minimal coordination between agents via shared /tmp/ report files. Agents don't call each other directly — they check for reports from other agents and note findings. Prevents circular dependencies.
Overseer Panel TEMPLATE
Multi-model architecture review that runs Gemini + OpenAI + Grok + Opus in parallel. Each model brings a different perspective. Consensus scoring: UNANIMOUS (all agree) > MAJORITY (2/3) > SINGLE. Grok plays devil's advocate.
Delegation Enforcement TEMPLATE
Hook-based rule that prevents the orchestrator from editing code directly. All code changes must go through specialized agents (e.g., @local-coder, @debug). Enforced by PreToolUse hooks that block Edit/Write on code files.
Guard Enforcement TEMPLATE
Safety limits checked after every iteration: maxIterations (stop loops), maxCostUSD (budget cap), wallClockMinutes (timeout), sameError3x (force escalation), verificationFailed (rollback). All guards enforced simultaneously.
One Task Per Session TEMPLATE
Rule that each Claude session should handle exactly one task. After completion: commit, start fresh. Prevents context rot and ghost code. "Session Hygiene" principle.
🌐 Industry Standards & Ecosystem (Feb 2026)
The emerging standards, frameworks, and tools shaping how AI agents are configured industry-wide
Two-Pack Architecture STANDARD
The emerging design pattern (Feb 2026) of separating agent configuration into two independent packs: Agent Pack (who the agent IS — identity, portable across projects) and Project Pack (what the project IS — context, portable across agents). Combined at runtime: Agent Pack (stable prefix) + Project Pack (variable per project) + Task (per-task).
Runtime assembly: Agent Pack → Project Pack → Task = Full Context Google calls this: "stable prefix + variable suffix"
Researched: Feb 13, 2026 | See: Handbook Ch 8
Agent Pack STANDARD
The agent's identity package — personality, goals, relationships, memories, skills, guardrails, voice/style. Portable: the same agent can work on different projects while staying "themselves." In PIA: Soul JSON. In industry: SOUL.md.
PIA equivalent: src/souls/personalities/*.json | Industry: SOUL.md
Project Pack STANDARD
Project-specific instructions — tech stack, file structure, coding conventions, build commands, architecture, testing rules, project status. Portable: different agents can join the same project by reading the Project Pack.
PIA equivalent: DAO_AGENT_BRIEFING.md | Industry: CLAUDE.md or AGENTS.md
AGENTS.md STANDARD
Open Markdown standard for guiding AI coding agents. A "README for agents" — project context, build commands, code style, architecture, security. First released Aug 2025 by OpenAI. Now governed by the Linux Foundation's Agentic AI Foundation (AAIF). 60,000+ repos use it. Default size limit: 32 KiB (OpenAI Codex). Supported by: Cursor, Codex, GitHub Copilot, Apple Xcode 26.3, Google Jules, Windsurf, and 20+ tools.
// ~/.codex/AGENTS.md (global defaults) // repo-root/AGENTS.md (project-level) // services/payments/AGENTS.override.md (directory override) // Discovery: closer files override farther ones
Spec: agents.md | Note: Claude Code uses CLAUDE.md instead (can symlink)
CLAUDE.md STANDARD
Anthropic's project configuration file for Claude Code. Persistent memory injected at session start. 7 recommended sections: Project Overview, Tech Stack, Commands, Workflows, Code Style, Architecture, Terminology. Auto-generated with /init. Every token loads every request — keep concise (~500 lines, ~8K tokens recommended). Hierarchy: ~/.claude/CLAUDE.md (global) → repo root (project) → subdirectories (scoped).
// Minimum viable CLAUDE.md: // 1. One-sentence project description // 2. Package manager (if not npm) // 3. Non-standard build/test commands
See: Dometrain Guide
SOUL.md Framework STANDARD
Created Feb 2, 2026 — brand new (11 days old). Defines WHO an agent is in Markdown. Three core files: SOUL.md (identity: values, boundaries, principles), STYLE.md (voice: tone, patterns, communication), SKILL.md (capabilities: operating instructions). Plus data/ folder for source material and examples/ for calibration (good vs bad outputs). Philosophy: specificity beats vagueness, include contradictions, test and iterate.
your-soul/ SOUL.md → Core Truths + Boundaries + The Vibe STYLE.md → Voice patterns + tone + how you write SKILL.md → Operating instructions + capabilities data/ → Writing, influences, source material examples/ → good-outputs.md + bad-outputs.md
GitHub: aaronjmars/soul.md | Community: souls.directory
Soul Stack STANDARD
The complete SOUL.md file structure. Includes: BUILD.md (agent uses to construct the soul), SOUL.md (identity), STYLE.md (voice), SKILL.md (capabilities), data/ folder (raw source: writing, tweets, influences), examples/ folder (good-outputs.md, bad-outputs.md for calibration). Three options to build: /soul-builder (AI interview), from your data, or manual template fill.
Instruction Budget STANDARD
The practical limit on how many instructions an agent can reliably follow. Frontier LLMs can handle ~150-200 instructions with reasonable consistency. Smaller models handle fewer. Every instruction loads on every request — this is a hard budget constraint regardless of task relevance. Coined by Kyle at HumanLayer.
Source: HumanLayer Blog
Context Engineering STANDARD
The discipline of managing what goes into an agent's context window. Encompasses: choosing what information to include/exclude, ordering it for maximum impact, keeping it fresh, and measuring its effect on agent performance. Martin Fowler's hierarchy: user-level config → project-level config → directory-level overrides → task-specific instructions.
Source: Martin Fowler
Progressive Disclosure STANDARD
Best practice for organizing agent instructions hierarchically. Root AGENTS.md/CLAUDE.md contains essentials only (project description, package manager, build commands). Domain-specific rules live in nested files (docs/TYPESCRIPT.md, services/payments/AGENTS.md). Agent gets progressively more detail as it enters specific directories. Prevents bloated root configs.
Discovery Order STANDARD
How AI coding tools find and merge configuration files. Typical pattern: starts at global config (~/.codex/ or ~/.claude/) → walks from Git root to current directory → at each level checks for override file first, then standard file. Files concatenate root-down; closer files override earlier guidance. Codex: AGENTS.override.md > AGENTS.md at same level.
Agentic AI Foundation (AAIF) STANDARD
Linux Foundation organization governing the AGENTS.md standard (as of Dec 2025). Co-founded by OpenAI, Anthropic, and Block. Supported by Google, Microsoft, AWS, Bloomberg, Cloudflare. Also oversees MCP (Anthropic's Model Context Protocol) and goose (Block's open-source agent). First major cross-company AI agent standards body.
Model Context Protocol (MCP) STANDARD
Anthropic's standard for connecting AI agents to external tools and services. Like USB for AI — any tool that speaks MCP can plug into any agent that supports MCP. Enables: file systems, databases, APIs, web browsers, and custom tools. Now part of AAIF under the Linux Foundation. Supported by Claude Code, OpenAI Codex, Google ADK, Apple Xcode 26.3, and many more.
Context7 STANDARD
An MCP server that injects up-to-date library documentation into an agent's context window. Solves the "stale training data" problem — when you're using Next.js 15 or React 19, Context7 pulls current docs. Tools: resolve-library-id + get-library-docs. NOT for agent configuration — purely for library/framework documentation retrieval. Useful alongside our soul system, not a replacement.
GitHub: upstash/context7
Google ADK STANDARD
Google's Agent Development Kit — open-source framework for building AI agents. Agent config: name (required), model (required), description (recommended for multi-agent), instruction (critical — personality + constraints + tool usage), tools (list of capabilities). Supports YAML config, dynamic instructions via session state, parent-child agent relationships, and multi-agent orchestration.
// agent.yaml name: assistant_agent model: gemini-2.5-flash instruction: | You are a helpful assistant that...
Docs: google.github.io/adk-docs
Agent Teams STANDARD
Claude Opus 4.6 experimental feature (Feb 5, 2026). Multiple agents collaborate on different parts of tasks simultaneously. Each agent can have its own identity (via subagent AGENTS.md files) while sharing project context. Represents the next evolution of multi-agent orchestration — similar to PIA's Fisher2050/Ziggi/Eliyahu team but built into the runtime.
souls.directory STANDARD
Community directory of SOUL.md personality templates. Launched Feb 2026. Open-source — browse, use, and publish agent personalities. A "soul store" for pre-built personality templates. Can be used as starting points for creating new agents.
Website: souls.directory | GitHub: thedaviddias/souls-directory
🌎 The 7 Layers of Agent Programming
Everything an agent receives before it starts working

Layer 1: Personality

Who the agent IS — communication style, traits, quirks. Makes each agent unique and consistent across sessions.

~875 tokens per soul

Layer 2: Goals

What the agent CARES about — 4-6 specific objectives that guide priorities and decision-making.

~200 tokens per soul

Layer 3: Relationships

Who the agent KNOWS — dynamics with other souls and humans. Affects tone and collaboration.

~300 tokens per soul

Layer 4: System Prompt

What the agent CAN DO — operational instructions, available tools, step-by-step procedures, boundaries.

~250-300 tokens per soul

Layer 5: Memories

What the agent HAS LEARNED — 20 recent + 10 high-importance memories from past work. Dynamic, grows over time.

~2,100 tokens (variable)

Layer 6: Config

HOW the agent operates — timing, thresholds, feature flags, quality minimums, scheduling.

~50 tokens per soul

Layer 7: Briefing Document

THE BIG PICTURE — full project context, tech stack, schemas, build plan, rules. Optional, for complex tasks.

~4,500 tokens (optional)
