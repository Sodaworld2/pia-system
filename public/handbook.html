The 7 Layers of Agent Programming

Layer 1: Personality REQUIRED

What: Multi-paragraph description of how the agent thinks, communicates, and behaves.

MetricMinimumSweet SpotMaximum
Length200 chars800-2,000 chars10,000 chars
Traits listed35-712
Quirks listed13-58
Style bullets24-610
Expert tip (SOUL.md): Specificity beats length. "Ends updates with confidence %" is better than "Is very analytical and detail-oriented." Real quirks and phrases anchor personality better than adjectives.

Layer 2: Goals REQUIRED

What: Specific objectives the agent actively pursues during every task.

MetricMinimumSweet SpotMaximum
Number of goals24-610
Words per goal58-1530
Expert tip (OpenAI): Goals should be actionable and measurable. "Run daily reviews and flag overdue items" is better than "Help manage projects." Each goal should answer "what would I see this agent DOING?"

Layer 3: Relationships RECOMMENDED

What: How the agent relates to other agents and humans. Affects tone and behavior.

MetricMinimumSweet SpotMaximum
Number of relationships1 (boss)3-510
Words per relationship1015-3050
Best practice: Always define the relationship with Mic (boss/owner). Then add relationships with other active agents. Include the dynamic, not just the role: "The Architect. Triggers Ziggi for code reviews. Respects Ziggi's thoroughness."

Layer 4: System Prompt REQUIRED

What: Operational instructions — what tools are available, what steps to follow, what boundaries to respect.

MetricMinimumSweet SpotMaximum
Total length200 chars500-2,000 chars5,000 chars
Procedure steps35-715
Tool/resource list24-612
Don't repeat personality here. The system prompt is about WHAT TO DO, not WHO YOU ARE. Personality goes in Layer 1.

What: Accumulated experience from past work. Auto-managed by the memory system.

Expert tip: Quality > quantity. A memory that says "PowerShell uses semicolons not && for chaining commands" (importance: 8) is worth more than 10 memories about routine file edits (importance: 3).

Layer 6: Config OPTIONAL

What: Timing, thresholds, and behavior switches.

MetricMinimumSweet SpotMaximum

How to Create an Agent (Step by Step)

Step 1: Define the Soul JSON

Create a new file in src/souls/personalities/your-agent.json:

// src/souls/personalities/your-agent.json { "id": "your-agent", "name": "YourAgent", "role": "Your Role Here", "email": "your-agent@sodaworld.com", "personality": "You are YourAgent, the [Role]... (200-2000 chars)", "goals": [ "Goal 1 — specific, actionable", "Goal 2 — measurable outcome", "Goal 3 — what you'd see them doing", "Goal 4 — with clear scope" ], "relationships": { "Mic": "The boss. Ultimate decision-maker.", "Fisher2050": "The PM. Coordinates with for task tracking." }, "system_prompt": "When processing a task:\n1. Step one\n2. Step two\n...", "config": { "scheduleTime": "09:00", "maxItems": 10 } }

Step 2: Register in seed-souls.ts

Add your new soul to the seed list so it loads on startup:

// src/souls/seed-souls.ts — add to the array: const soulFiles = [ 'fisher2050.json', 'ziggi.json', 'eliyahu.json', 'your-agent.json' // <-- add this ];

Step 3: Restart PIA

The new soul will be automatically loaded into the database:

npm run dev // Console: "Seeded soul: your-agent (YourAgent)"

Step 4: Verify via API

curl http://localhost:3000/api/souls/your-agent // Should return your soul definition with all fields

Step 5: Test with a Task

curl -X POST http://localhost:3000/api/orchestrator/run \ -H "Content-Type: application/json" \ -d '{ "task": "List all files in the current directory and describe what you see", "soulId": "your-agent", "maxTurns": 5, "maxBudgetUsd": 0.50 }'

Step 6: Seed Initial Memories (Optional)

curl -X POST http://localhost:3000/api/souls/your-agent/memories \ -H "Content-Type: application/json" \ -d '{ "category": "experience", "content": "Set up the PIA system on Machine #1", "importance": 7 }'

Alternative: Use the Agent Generator

Open /agent-generator.html to fill in a form and download the JSON file. Then place it in src/souls/personalities/.

Memory System Guide

Memory Categories — When to Use Each

CategoryWhen to UseExampleTypical Importance
experienceSomething the agent DID"Fixed 5 failing tests by updating mock library"5-8
decisionA choice was MADE"Chose TypeScript over Python for backend"7-9
learningSomething DISCOVERED"Snapshot tests break if regenerated without review"6-8
interactionCommunication HAPPENED"Fisher2050 requested code review for DAO changes"4-6
observationSomething NOTICED"CI pipeline slow on Fridays — batch jobs overlap"3-6
goal_progressProgress on OBJECTIVES"Completed 3/5 routing refactor milestones"5-7
summaryAUTO-GENERATED only"Feb 5-7: Fixed auth/session bugs, 5 improvements"5

Importance Scoring Guide

ScoreMeaningLifetimeExamples
9-10Critical / BreakthroughFOREVERMajor architecture decision, critical bug found, production incident
7-8Important / SignificantFOREVER (in "important" slot)Key technical choice, successful deployment, learned workaround
5-6Normal / StandardSummarized after 7 daysRoutine task completion, standard interaction, minor finding
3-4Low / MinorSummarized then prunedFile renamed, config tweaked, small observation
1-2Trivia / NoisePruned after 30 daysTemp file created, test output checked, minor log entry

Memory Lifecycle

Memory Created → Active (loaded in prompts) → 7+ days: Summarized → 30+ days + low importance: Pruned
High-importance memories (7+) are NEVER pruned. They stay in the "important memories" slot forever, giving the agent permanent knowledge of critical decisions and learnings.

Do's and Don'ts — Best Practices

Agent Design — DO

  • Be specific with personality — use real quirks, phrases, patterns
  • Keep goals actionable — "Run daily reviews" not "Be helpful"
  • Define relationships with dynamics, not just names
  • Separate personality (who) from instructions (how)
  • Include 3-5 signature phrases that anchor behavior
  • Add good/bad output examples for calibration
  • Name real influences (books, frameworks, people)
  • Test with a small task before giving complex work

Agent Design — DON'T

  • Don't use vague traits ("helpful", "smart", "efficient")
  • Don't write novel-length personality descriptions
  • Don't give more than 8 goals — focus beats breadth
  • Don't repeat personality in the system prompt
  • Don't always inject briefing docs for simple tasks
  • Don't set all memory importance to 10
  • Don't skip the Zero Questions Policy for autonomous agents
  • Don't forget to define escalation — when should the agent STOP?

Memory — DO

  • Score importance honestly — 7+ means it matters in 2 weeks
  • Use the right category — decision vs experience vs learning
  • Keep memory content concise — 100-300 chars is ideal
  • Seed critical memories when creating a new agent
  • Run summarization weekly for active agents
  • Prune monthly to prevent database bloat

Memory — DON'T

  • Don't store entire file contents as memories
  • Don't mark routine tasks as importance 9-10
  • Don't skip the "decision" category — decisions are gold
  • Don't let memories accumulate without summarization
  • Don't delete high-importance memories manually
  • Don't load 50+ memories into every prompt

Autonomy — DO

  • Set Zero Questions Policy for build agents
  • Set maxBudget and maxTurns on every task
  • Block dangerous commands (rm -rf, mkfs, etc.)
  • Require reading files before editing
  • Require tests after code changes
  • Log all actions for auditability

Autonomy — DON'T

  • Don't give unlimited budget — always cap spend
  • Don't skip guardrails "just this once"
  • Don't let agents run without timeout limits
  • Don't give file-write access without path restrictions
  • Don't forget same-error-3x escalation
  • Don't trust agent output without verification for critical ops

The Two-Pack Architecture

The Core Insight (Feb 2026)

The industry is converging on separating agent configuration into two independent packs. This is the single most important design pattern for agent systems in 2026.

Breaking news (this week): SOUL.md was created Feb 2, 2026 — just 11 days ago. AGENTS.md was donated to the Linux Foundation's Agentic AI Foundation (AAIF) in Dec 2025, co-founded by OpenAI, Anthropic, and Block. 60,000+ repos now have AGENTS.md. Apple Xcode 26.3 (Feb 3, 2026) added agentic coding support with AGENTS.md and CLAUDE.md. The two-pack separation is happening RIGHT NOW.

Pack 1: Agent Pack (SOUL.md)

"Who am I?"

Identity, personality, goals, relationships, memories, skills, guardrails, voice/style.

Portable: Same agent can work on different projects. Fisher2050 stays "Fisher2050" whether working on PIA, Videohoho, or a new project.

Industry names: SOUL.md, Agent Config, Persona, Agent Definition

Pack 2: Project Pack (CLAUDE.md / AGENTS.md)

"What am I working on?"

Tech stack, file structure, coding conventions, build commands, architecture, testing rules, project status.

Portable: Different agents can join the same project. A new agent can onboard by reading the Project Pack.

Industry names: CLAUDE.md, AGENTS.md, Project Config, Repo Instructions

Runtime Assembly

Agent Pack (stable prefix) → Project Pack (per-project) → Task (per-task) = Full Context

Google calls this "stable prefix + variable suffix." The Agent Pack rarely changes; the Project Pack changes per project; the Task changes per task.

SOUL.md: The Soul Stack (Created Feb 2, 2026)

Brand new framework — only 11 days old. Defines WHO an agent is.

// The Soul Stack file structure your-soul/ BUILD.md // Agent uses this to build your soul SOUL.md // Identity: values, boundaries, principles STYLE.md // Voice: tone, patterns, how you write SKILL.md // Operating instructions: what you can do data/ writing/ // Articles, posts, essays (source material) influences.md // Who shaped your thinking examples/ good-outputs.md // Calibration: this is right bad-outputs.md // Calibration: this is wrong

Three core sections: Core Truths (values, principles), Boundaries (guardrails), The Vibe (tone, communication style).

SOUL.md vs PIA Souls: Very similar concepts! SOUL.md uses markdown files; PIA uses JSON stored in SQLite. SOUL.md separates identity/voice/skills into 3 files; PIA combines them in one JSON. Both share the same philosophy: specificity beats vagueness, include contradictions, test and iterate.

New: souls.directory — Community Soul Templates

Launched Feb 2026. Open-source directory of SOUL.md templates. Browse, use, and publish agent personalities. Think of it as a "soul store" — you can find pre-built personality templates for different agent types.

What this means for PIA: Our Soul JSON = Agent Pack. Our Briefing Document = Project Pack. We're already using the two-pack pattern — we just need to formalize the Project Pack side (create a CLAUDE.md for the main repo, standardize briefing doc structure). We should also consider adding STYLE and SKILL separation like SOUL.md does. We are ahead of the curve — most tools don't have persistent agent identity yet. PIA has had it since Phase 1.
